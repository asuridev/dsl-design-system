orders core

Manages the full order lifecycle from cart creation through checkout, payment confirmation, delivery tracking and order completion. Validates prices against catalog via HTTP at checkout (monetary snapshot — OWASP A04). Maintains a local read model of customer delivery addresses fed by CustomerAddressUpdated events from the customers BC. Coordinator of the CheckoutSaga.

UC-ORD-005 Checkoutscaffold customer command POST /carts/{cartId}/checkout Cart.checkout ORD-RULE-001ORD-RULE-002 —

Performs checkout of the customer's cart. Validates prices against the catalog (HTTP call — OWASP A04 monetary snapshot), transitions the cart to CHECKED_OUT, and creates a new PENDING_PAYMENT Order with frozen price snapshots. Emits OrderPlaced to trigger the CheckoutSaga (stock reservation → payment). The selected delivery address must exist in the CustomerAddress local read model and must not be marked as deleted.

Flujos (Given / When / Then)
FL-ORD-009 · Checkout — happy path (pago con tarjeta)

Given:

  • Carrito OPEN con 2 ítems: { productId: "p1", quantity: 2 }, { productId: "p2", quantity: 1 }.
  • CustomerAddress con addressId = "addr-1", deleted = false, del cliente autenticado.
  • Catalog devuelve: p1 = ACTIVE, price = 15000 COP; p2 = ACTIVE, price = 8500 COP.

When:

  • POST /api/orders/v1/carts/{cartId}/checkout con:
  { "deliveryAddressId": "addr-1" }

Then:

  • HTTP 201 Created
  • Header Location: /api/orders/v1/orders/{newOrderId}
  • El carrito transiciona a CHECKED_OUT.
  • Se crea una Order con status = PENDING_PAYMENT:
  • paymentMethod = CARD
  • total = { amount: 38500, currency: COP } (2×15000 + 1×8500)
  • deliveryAddress = snapshot de la dirección addr-1
  • 2 OrderLine: precios congelados del catálogo.
  • Se emite OrderPlaced al outbox con las líneas y el total.

catalog customer-supplier http validateProductsAndPrices UC-ORD-005 Checkout internal-jwt — Mandatory synchronous HTTP call to catalog at checkout time to obtain the authoritative price for each product in the cart. Mandatory HTTP for monetary integrity — prevents OWASP A04 fraud via stale or client-manipulated prices. The customer never sends prices; prices are always fetched from the catalog. Mandatory HTTP for monetary snapshot (OrderLine.unitPrice reads Product.price). Designer evaluated LRM trade-offs including OWASP A04 fraud risk and chose HTTP. Price is always read from catalog in real time at checkout — no inconsistency window. The customer sends only productId + quantity; the backend reads the authoritative price.

notifications event message-broker OrderConfirmedOrderCancelled — — — Notifications BC sends an order confirmation message when the order is confirmed and a cancellation notice when the order is cancelled.
payments event message-broker OrderCancelled — — — Payments BC consumes OrderCancelled to check if a card payment was already captured. If so, it initiates a refund via payment-gateway and emits PaymentRefunded. This closes the compensation chain for post-confirmation cancellations that occur outside the CheckoutSaga (e.g., admin cancellation, customer cancellation request).
